# ApplyOS launch implementation plans

Generated by the `improve` skill on 2026-07-16 and reconciled on 2026-07-17
against commit `3841724` (ApplyOS v0.9.0). Plan 000 was completed by that commit:
schema v5, profile isolation, semantic JavaScript checking, fail-closed CI
browser verification, and the expanded regression suite are now the baseline.
The existing `plans/` directory contains a product roadmap diagram, so these
executor-ready plans live in `advisor-plans/` to avoid mixing purposes.

These plans answer the six requested launch requirements. Requirement 3 is
split into a database-foundation plan and a sync/recruiter-boundary plan because
combining schema, authorization, migration, offline sync, and conflict handling
into one implementation task would be too risky.

These files are planning artifacts, not automatic repository instructions. An
executor must act on a plan only after the maintainer explicitly selects it.

## Recommended architecture

Use Supabase for the first cloud version: LinkedIn OIDC through Supabase Auth,
Postgres with Row Level Security, a private Storage bucket for opted-in resume
sync, and Edge Functions for privileged operations such as support-report
submission and account deletion. Keep `chrome.storage.local` as the offline
working cache. The extension must remain useful without a network connection,
and users must explicitly opt into cloud sync.

LinkedIn login is authentication only. It provides a lite identity profile; it
does not import a user's LinkedIn contacts, network, messages, work history, or
recruiter permissions. It must not be marketed as identity verification.

## Delivery sequence

1. **Completed launch gate:** Plan 000 landed in `3841724`. Re-run its four
   commands at the start and end of every remaining plan.
2. **Week 1 — decisions and cloud foundation:** execute 001, 002, and 007;
   create staging and production Supabase projects; publish a privacy policy,
   terms, support page, and data-deletion instructions.
3. **Week 2 — account:** execute 003 and complete the LinkedIn developer-app
   configuration. LinkedIn product access is an external dependency and should
   be requested at the start of Week 1.
4. **Weeks 3–4 — local-first sync:** execute 004. Do not proceed to production
   migration until first-sync, account-switch, offline, conflict, export, and
   deletion tests pass.
5. **Week 4 — support reporting:** execute 006 after authenticated Edge
   Functions are available. It can run in parallel with the latter half of 004.
6. **Week 5 — tutorial and launch hardening:** execute 005, then run the full
   extension verification suite, staging OAuth smoke tests, privacy-copy review,
   and Chrome Web Store disclosure review.

A realistic remaining estimate for one experienced engineer is **5–7 weeks**, excluding
LinkedIn approval and Chrome Web Store review time. Compressing the work should
reduce scope, not skip RLS, migration, deletion, or privacy tests.

## External setup checklist

- Create separate staging and production Supabase projects.
- Resolve the working product name and run qualified naming/trademark clearance
  before registering production OAuth apps, domains, policies, or billing
  descriptors. Treat the current ApplyOS name as provisional until that review
  is complete.
- Create a LinkedIn developer app tied to the ApplyOS company page and request
  the **Sign In with LinkedIn using OpenID Connect** product.
- Register the Supabase callback URL in LinkedIn and the Chrome
  `https://<extension-id>.chromiumapp.org/auth/callback` URL in Supabase's
  redirect allow list.
- Keep the LinkedIn client secret and all Supabase service-role credentials in
  provider dashboards/server secrets only. Never put them in the extension,
  source control, build output, logs, or tests.
- Obtain a production extension ID before final redirect configuration; the
  Chromium callback host is derived from that ID.
- Publish accurate privacy, retention, export, and account-deletion policies
  before enabling sync for real users.
- Update Chrome Web Store disclosures because the current product copy says all
  data stays on-device and the cloud release introduces transmission/storage.

## Findings considered and rejected

- **Replace local storage with cloud-only persistence:** rejected. The existing
  autofill and dashboard paths are designed around immediate local access, and
  cloud-only storage would make core job-application behavior depend on network
  availability.
- **Use LinkedIn login to import contacts or the user's network:** rejected.
  LinkedIn OIDC supplies authentication and lite profile claims, not networking
  data access. Contacts remain manually entered unless a separately approved,
  clearly consented integration is designed later.
- **Let recruiters query private profiles, applications, contacts, answers, or
  notes:** rejected. Recruiter discovery must use a separate candidate-approved
  publication table. Private job-search activity is never recruiter-visible.
- **Allow anonymous direct inserts into the support table:** rejected. A public
  database insert policy is abuse-prone. Reports go through a validating,
  rate-limited Edge Function, with copy/download fallback when signed out.
- **Add Stripe or paid AI now:** rejected because requirement 6 explicitly asks
  for a README TODO only. Billing begins only after the product and pricing model
  are approved.
